11 # Related work
12 
13 Jnif uses a clean object model to represent the java class files. Modyfing the
14 class files is just matter of modifying the object models and then serializer
15 and deserializer to convert to a memory buffer and send to the jvm ready to
16 execution.
17 
18 Ability to **instrument** the JVM by only adding a small agent.
19 
20 
21 Usually to instrument the *JVM* another JVM is needed.
22 To instrument regular applications this is just fine,
23 but when you need to instrument the java library itself you encounter
24 bootstrap problems.
25 That is why a native approach is more accurate.
26 
27 Instrumentation and client code are more coupled, this is how it should be.
28 
29 It allows to simplify the development because you do not need a TCP
30 connection to instrument code on other JVM process.
31 Lowoverhead agent.
32 Modular parser, you pay what you need. Full power of templates for better
33 performance.
34 
35 
36 Since the instrumentation should be really really fast, and therefore having a
37 negligent overhead,
38 * it could be added in production environments?
39 * and also it will be more benefitial in debuggers and profilers where you may
40 instrument on demand.

42 # Development
43 
44 Goals:
45 * Modular parser/writer using templates. no more modular!
46  Each attribute kind can be plugged into the main parser.
47 * Zero-memory footprint.
48  The API is built so that it uses zero dynamic memory, i.e., no calls to
49  malloc/new! to gain more performance and memory consumption.
